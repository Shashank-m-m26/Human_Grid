# Agent Spec

## Overview
HumanGrid is a production-quality multi-agent enterprise application for the Kaggle AI Agents Intensive Vibe Coding Capstone in the "Agents for Business" track.

The system acts as an enterprise AI concierge. A user describes a mission in natural language, and the application coordinates seven specialized agents to interpret the mission, break it into tasks, identify required skills, discover employees, rank candidates, coordinate assignments, and track execution through completion.

Internally, the mission intake flow is implemented as the Mission Engine to reflect the full lifecycle from intake through orchestration while preserving the existing external mission-first experience.

The implementation must preserve the playbook exactly:

- Frontend: Next.js, TailwindCSS, shadcn/ui, Framer Motion
- Backend: Python, FastAPI, Google ADK
- Model: Gemini 2.5 Flash
- Knowledge storage: JSON files
- MCP: Python MCP SDK
- Auth: simple login
- Theme: dark futuristic enterprise dashboard

## Example Use Cases
- "I need an MCP expert before tomorrow."
  Expected outcome: mission is structured, tasks are planned, skill requirements are identified, candidate employees are ranked, and the best expert is recommended with reasoning and confidence.
- "I need reviewers for my client demo."
  Expected outcome: reviewers are selected based on availability, collaboration score, trust score, and prior project fit.
- "I need a DevOps mentor."
  Expected outcome: mentor candidates are discovered, ranked, and surfaced with workload-aware explanations.
- "I need a cross-functional hackathon team."
  Expected outcome: a multi-role team is assembled across departments and tracked in the mission dashboard and Mission Command Center.

## Constraints & Safety Rules
- Do not redesign the project.
- Do not rename agents.
- Do not replace JSON with SQL.
- Do not collapse the multi-agent workflow into a single agent.
- Only the People Discovery Agent and Coordinator Agent should invoke MCP tools directly unless otherwise required.
- Implement prompt injection protection, input validation, audit logging, secure JSON access, basic authentication, and error handling.
- Avoid automated browser UI tests or automated multi-turn tests against real LLM endpoints.

## Architecture & Sub-Agents
- Root ADK Agent
  - Orchestrates all seven agents and preserves role boundaries.
  - Drives the Mission Engine state transitions and real-time agent execution timeline.
- Intent Agent
  - Extracts mission information and produces structured mission JSON.
- Mission Planner Agent
  - Converts mission into executable tasks.
- Skill Mapper Agent
  - Maps tasks to required skills.
- People Discovery Agent
  - Searches the complete enterprise knowledge graph through MCP tools.
  - Discovery scope includes employees, projects, departments, and skills rather than employee records only.
- Intelligence Agent
  - Ranks candidates using skills, experience, previous projects, collaboration score, trust score, workload, and availability.
  - Produces explainable recommendations with confidence score, concise explanation, and reasoning for every selected employee.
- Coordinator Agent
  - Assigns employees, creates missions, schedules meetings, and sends notifications through MCP.
- Mission Tracker Agent
  - Tracks lifecycle updates and dashboard state.

## Frontend Experience Requirements
- Preserve the non-chat mission-first landing page.
- Add a Mission Command Center page to serve as the primary operations view after mission submission.
- Add an Agent Execution Timeline component that visualizes every AI agent in real time.
- Continue surfacing mission timeline, recommended team, activity feed, mission progress, and intelligence explanation.

## Success Criteria
- User can create a mission using natural language.
- Backend returns a structured mission and planned task breakdown through the Mission Engine flow.
- Agent pipeline is represented explicitly and keeps responsibilities separated.
- Candidate recommendations include explainable ranking factors, confidence score, and reasoning.
- Mission dashboard and Mission Command Center can represent progress, timeline, activity feed, team, and intelligence explanation.
- Enterprise data model supports approximately 100 realistic employees across required departments.

## Non-Functional Requirements
- Modular architecture with separate frontend, backend, agents, MCP, and JSON data layers.
- Production-ready structure with consistent naming and reusable utilities.
- Dark enterprise UI that does not resemble a chatbot.
